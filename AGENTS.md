# Repository Agent Guidance

이 저장소는 여러 Codex 기반 업무 자동화 에이전트를 설계하고 실험하기 위한 public-safe skeleton입니다.

가장 중요한 폴더는 `agents/00-agent-harness-translator`입니다. 새 에이전트 아이디어는 먼저 0번 에이전트의 템플릿으로 번역한 뒤 개별 에이전트 폴더에 반영합니다.

## 기본 원칙

- 실제 개인정보, 고객자료, 계약서, 급여자료, 세금자료, 내부 문서, 네이버웍스 토큰, 쿠키, API 키를 커밋하지 않습니다.
- 외부 서비스 연동은 기본적으로 dry-run 또는 draft mode로 설계합니다.
- 메시지 발송, 이메일 발송, 파일 업로드, 게시물 업로드, 크롤링은 사람의 명시 승인이 필요합니다.
- 에이전트 산출물은 초안으로 취급하고, 최종 제출/배포 전 사람이 검토합니다.
- 각 에이전트 폴더는 독립된 작은 제품처럼 다룹니다.
- Codex 런타임 수정이 필요한 아이디어는 제외합니다.
- subagent 기반 병렬 작업은 허용하지만, 별도 멀티에이전트 오케스트레이션이 필요한 아이디어는 제외합니다.
- 기본 목표는 로컬 Codex 환경에서 동작하는 에이전트입니다.

## 에이전트 폴더 규칙

각 에이전트 폴더는 최소한 다음 파일을 둡니다.

- `AGENTS.md`: 해당 에이전트의 역할, 입력, 출력, 작업 규칙
- `.codex.toml`: 상태, 도구, 승인 정책, 데이터 민감도 설정

선택적으로 다음 폴더를 둘 수 있습니다.

- `prompts/`: 시스템 프롬프트, 작업 프롬프트, 검토 프롬프트
- `templates/`: 문서 템플릿, 설문 템플릿, 메시지 템플릿
- `examples/`: 공개 가능한 샘플 입력/출력
- `scripts/`: API 호출, 문서 변환, 데이터 처리 스크립트
- `tests/`: 샘플 기반 검증
- `docs/`: 설계 메모, 운영 매뉴얼, 리스크 검토

## 작업 방식

1. 새 에이전트 아이디어는 먼저 `agents/00-agent-harness-translator/templates/harness-parts-spec.md`에 맞춰 번역합니다.
2. 먼저 해당 에이전트의 `AGENTS.md`와 `.codex.toml`을 읽습니다.
3. 현재 상태가 `idea`라면 구현보다 범위 정리와 리스크 검토를 우선합니다.
4. 민감정보나 외부 연동이 필요하면 샘플/모의 데이터로 대체합니다.
5. 산출물 생성 기능은 먼저 로컬 초안 생성으로 구현합니다.
6. 외부 발송/업로드 자동화는 별도 승인 흐름을 만든 뒤 구현합니다.

## 하네스 부품 구분

0번 하네스 번역기는 작업 아이디어를 아래 부품으로 분해합니다.

- 지시: 시스템 프롬프트, 입력 프롬프트
- 지식/상태: 메모리, skill
- 능력/행동: MCP, 서브에이전트
- 제어/횡단: hook

각 부품에는 입력, 출력, 실패 모드, 불변식이 있어야 합니다. 특히 실패 모드와 불변식이 비어 있으면 구현을 시작하지 않습니다.

## Public Repo Safety

이 저장소는 공개를 전제로 합니다. 파일을 추가하기 전에 다음을 확인합니다.

- 실제 회사/고객/직원/강사/기자 개인정보가 포함되어 있지 않은가
- 인증 토큰, 쿠키, 세션, 비밀번호, API key가 포함되어 있지 않은가
- 네이버카페, 네이버웍스, 이메일, 메신저에서 가져온 원문을 저장하지 않았는가
- 저작권이 있는 제안서/과업지시서/입찰 문서를 그대로 넣지 않았는가
- 자동 배포나 업로드가 즉시 실행되는 코드가 아닌가
