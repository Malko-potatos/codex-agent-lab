# 하네스 부품 선택 Decision Tree

이 문서는 업무 아이디어를 Codex 하네스 부품으로 나눌 때의 판단 기준입니다. 새 에이전트를 설계할 때는 먼저 가장 작은 로컬 초안 흐름을 만들고, 필요할 때만 외부 도구와 제어 장치를 추가합니다.

## 빠른 판정표

| 질문 | 예 | 아니오 |
|------|----|--------|
| 매번 지켜야 하는 정체성, 금지사항, 출력 계약입니다. | 시스템 프롬프트 | 다음 질문 |
| 이번 실행마다 바뀌는 목표, 파일, 날짜, 파라미터입니다. | 입력 프롬프트 | 다음 질문 |
| 반복 가능한 정적 절차나 도메인 지식입니다. | skill | 다음 질문 |
| 작업 사이에 보존되어야 하는 선호, 결정, 진행 상태입니다. | memory | 다음 질문 |
| 파일, 브라우저, 외부 API, 사내 시스템처럼 밖의 자원을 조작합니다. | MCP 도구 | 다음 질문 |
| 병렬 조사, 독립 검토, 역할 분담처럼 별도 작업자에게 맡기는 편이 좋습니다. | subagent | 다음 질문 |
| 도구 실행 전후에 승인, 차단, 감사, 마스킹이 필요합니다. | hook | 일반 프롬프트/문서 규칙 |

## 시스템 프롬프트로 둘 것

다음 조건이면 시스템 프롬프트에 둡니다.

- 모든 실행에서 유지되어야 하는 역할입니다.
- 법적/보안/품질 제약처럼 사용자가 바꿔도 깨면 안 되는 규칙입니다.
- 출력 형식과 검토 절차가 에이전트의 정체성을 이룹니다.

피해야 할 것:

- 특정 고객명, 날짜, 파일명 같은 실행별 정보
- 사용자가 매번 바꿀 수 있는 톤이나 분량
- 외부 도구의 인증값

## 입력 프롬프트로 둘 것

다음 조건이면 입력 프롬프트에 둡니다.

- 이번 작업의 목적과 범위입니다.
- 이번 실행에서만 쓰는 파일, 조건, 기간, 대상입니다.
- 누락되면 질문해야 하는 파라미터입니다.

피해야 할 것:

- 에이전트가 절대 위반하면 안 되는 안전 규칙
- 반복 업무 절차 전체
- 장기 보존해야 하는 운영 결정

## Skill로 둘 것

다음 조건이면 skill 후보로 둡니다.

- 여러 에이전트나 여러 실행에서 반복되는 절차입니다.
- 문서 작성법, 검토법, 변환법처럼 순서가 중요합니다.
- 버전 관리되는 정적 지식으로 유지할 수 있습니다.

예:

- 제안서 요구사항 대응표 작성 절차
- 결과보고서 사진 캡션 작성 규칙
- 만족도조사 문항 편향 점검 절차

불변식:

- skill은 실제 운영 데이터나 인증정보를 담지 않습니다.
- 오래된 절차일 수 있음을 표시하고 검토 날짜를 둡니다.

## Memory로 둘 것

다음 조건이면 memory 후보로 둡니다.

- 사용자의 선호, 이전 결정, 진행 상태처럼 다음 작업에 영향을 줍니다.
- 실행 결과에서 새로 알게 된 사실입니다.
- 매번 전체 파일을 읽지 않고도 이어가기 위한 상태입니다.
- 공식 Codex memory가 켜져 있을 때 local recall layer로 도움이 됩니다.

예:

- 특정 에이전트의 우선순위 결정
- 이미 검토한 외부 연동의 승인 범위
- 사용자가 선호하는 문서 문체나 산출물 형식

불변식:

- 민감정보, 토큰, 쿠키, 고객 원문은 저장하지 않습니다.
- 오래되었을 가능성이 있는 기억은 현재 파일이나 사용자 확인으로 갱신합니다.
- 항상 적용해야 하는 규칙은 memory가 아니라 `AGENTS.md` 또는 checked-in docs에 둡니다.
- repo 안 임의 `MEMORY.md`를 공식 Codex memory 저장소처럼 취급하지 않습니다.

저장 위치:

- 공식 Codex memory: `~/.codex/memories/`
- repo에 둘 것: memory policy, public-safe 샘플, 반드시 적용할 규칙 문서
- repo에 두지 말 것: generated memories, 실제 업무 맥락, 민감 원문

## MCP 도구로 둘 것

다음 조건이면 MCP 도구 후보로 둡니다.

- 외부 시스템이나 로컬 시스템 자원에 접근해야 합니다.
- 같은 동작을 여러 번 안정적으로 호출해야 합니다.
- 입력/출력 schema와 권한 경계를 명확히 정의할 수 있습니다.

예:

- 네이버웍스 파일 검색
- Google Forms dry-run schema 생성
- 로컬 문서 템플릿 렌더링

불변식:

- 도구는 신뢰 경계 밖 자원에 마음대로 접근하지 않습니다.
- 쓰기/전송/업로드 동작은 별도 승인 전에는 dry-run입니다.
- 인증 실패나 권한 오류는 우회하지 않고 사용자에게 에스컬레이션합니다.

## Subagent로 둘 것

다음 조건이면 subagent 후보로 둡니다.

- 독립적으로 병렬 조사하거나 검토할 수 있습니다.
- 부모 에이전트가 작은 입력 계약을 주고 결과를 검증할 수 있습니다.
- 실패해도 부모가 전체 작업을 통제할 수 있습니다.

예:

- RFP 요구사항 추출과 기존 제안서 스타일 분석을 분리
- 보도자료 fact-check와 제목 후보 생성을 분리
- 신규사업 리서치에서 시장/경쟁사/운영 리스크를 병렬 검토

불변식:

- 자식에게 필요한 최소 컨텍스트만 전달합니다.
- 재귀 깊이와 작업 수를 제한합니다.
- 부모가 출처, 가정, 누락을 검증합니다.

## Hook으로 둘 것

다음 조건이면 hook 후보로 둡니다.

- 도구 실행 전후에 반드시 검사를 해야 합니다.
- 위험 행동을 차단하거나 승인 요청으로 바꿔야 합니다.
- 감사 로그, 민감정보 마스킹, 정책 검증이 횡단 관심사입니다.

예:

- `.env`, `credentials/`, `private/` 경로 커밋 차단
- 이메일 발송 도구 호출 전 승인 요구
- 외부 업로드 전 파일 민감도 확인

불변식:

- hook 자체가 실패했을 때 흐름을 통과시킬지 중단할지 정책이 있어야 합니다.
- 차단 메시지는 사용자가 복구할 수 있을 만큼 구체적이어야 합니다.
- hook은 비밀값을 로그에 남기지 않습니다.

## 설계 순서

1. 시스템 프롬프트와 입력 프롬프트로 작은 로컬 초안 흐름을 만듭니다.
2. 반복 절차가 보이면 skill로 분리합니다.
3. 작업 간 상태가 필요할 때만 memory를 둡니다.
4. 외부 자원 접근은 MCP 계약과 승인 경계를 먼저 씁니다.
5. 병렬성이 실제 시간을 줄일 때만 subagent를 둡니다.
6. 위험 행동이 보이면 hook으로 승인, 차단, 감사 정책을 설계합니다.
