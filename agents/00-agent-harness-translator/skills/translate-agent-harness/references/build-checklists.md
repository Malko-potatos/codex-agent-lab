# 에이전트 산출물 생성 체크리스트

이 문서는 0번 에이전트 하네스 번역기가 새 에이전트 폴더를 만들거나 기존 에이전트 폴더를 고칠 때 사용하는 검수표입니다.

## 공통 전제

- 공개 저장소에 실제 고객 문서, 원문 데이터, 인증정보, 개인정보를 넣지 않습니다.
- 모든 외부 전송, 업로드, 게시, 발송, 로그인 세션 사용은 기본값을 `dry-run`으로 둡니다.
- 산출물은 초안이며 최종 제출과 실행은 사람이 검토합니다.
- 실패 모드와 불변식이 비어 있으면 설계가 끝난 것으로 보지 않습니다.
- 0번 에이전트 폴더 내부의 지침, 설정, skill reference는 상위 폴더 상대경로를 참조하지 않습니다.
- 필요한 기준 문서는 이 skill의 `references/` 또는 `assets/` 안에 둡니다.

## Self-Contained 폴더 검증

0번 에이전트와 0번이 생성하거나 수정하는 에이전트는 자기 폴더만으로 핵심 작업을 수행할 수 있어야 합니다.

- `.codex.toml`의 `reference_docs` 값이 모두 해당 에이전트 폴더 안의 상대경로인지 확인합니다.
- `AGENTS.md`, `SKILL.md`, reference, asset 파일에 상위 폴더 상대경로가 없는지 확인합니다.
- 루트 `docs/`의 내용을 필요로 하면 그대로 참조하지 말고, 필요한 부분만 `references/`에 public-safe로 복제합니다.
- 새 reference를 추가하면 `SKILL.md`, `.codex.toml`, README 폴더 구조를 함께 갱신합니다.
- 완료 전 상위 폴더 상대경로 검색 결과가 비어 있는지 확인합니다.

## Workflow 상태 검증 체크리스트

0번 에이전트는 `AGENTS.md`나 `.codex.toml` 초안을 쓰기 전에 사용자의 시나리오가 workflow로 충분히 설명되는지 먼저 확인합니다.

## 페르소나 검증 체크리스트

0번 에이전트는 비개발자 사회인과 직장인을 대상으로 하는 파인만 학습식 강사입니다.

- 설명이 먼저 업무 예시나 쉬운 말로 시작하는지 확인합니다.
- `hook`, `skill`, `MCP`, `memory`, `AGENTS.md`, `.codex.toml`, `subagent` 같은 약속된 용어에 쉬운 설명이 붙었는지 확인합니다.
- 부족한 정보를 사용자의 실수처럼 말하지 않았는지 확인합니다.
- 질문마다 답이 없을 때 에이전트가 어디서 헷갈리는지 설명했는지 확인합니다.
- 수강생용 문서는 `~입니다`, `~합니다` 어투를 유지하는지 확인합니다.
- 기술 용어를 많이 나열해 권위적으로 보이게 만들지 않았는지 확인합니다.

### 1. 상태 식별

- `intake`, `scope_check`, `input_inventory`, `plan`, `draft`, `validate`, `approval`, `deliver`, `remember_or_close` 중 필요한 상태를 골랐는지 확인합니다.
- 제외한 상태가 있으면 왜 제외했는지 적었는지 확인합니다.
- 각 상태에 입력과 출력이 있는지 확인합니다.

### 2. 전이 조건

- 각 상태가 다음 상태로 넘어가는 조건을 적었는지 확인합니다.
- 필수 입력이 없을 때 멈추는지, 질문하는지, 샘플 데이터로 대체하는지 정했는지 확인합니다.
- 반복, 분기, 실패 후 재시도 경로가 있으면 표시했는지 확인합니다.
- 요구사항 디테일이 부족해도 대략적인 상태전이 다이어그램이 있는지 확인합니다.
- 다이어그램의 각 화살표에 skill, hook, MCP, `AGENTS.md`, `.codex.toml`, memory, subagent 중 어떤 부품이 전이를 만들거나 막는지 표시했는지 확인합니다.
- 사용자의 예상과 다르게 흘러가는 전이를 `예상과 다름`, `보완 필요`, `위험`, `차단` 중 하나로 표시했는지 확인합니다.
- 부족한 개발 요청을 임의의 기간 목표나 수업 단계 제한으로 바꾸지 않았는지 확인합니다.

### 3. 부족 정보 영향

- 부족 정보를 단순히 질문으로만 쓰지 않았는지 확인합니다.
- 부족한 정보가 어떤 상태나 전이를 약하게 만드는지 적었는지 확인합니다.
- 그 결과 사용자의 기대 workflow가 어떻게 달라질 수 있는지 설명했는지 확인합니다.
- 불분명한 정보 때문에 에이전트가 빠질 수 있는 상황을 적었는지 확인합니다.
- 설명이 비개발자 사회인이나 대학생도 이해할 수 있는 쉬운 말인지 확인합니다.
- 판정이 `충분`, `보완 필요`, `위험`, `차단` 중 하나인지 확인합니다.

### 4. 하네스 부품 보강

- 역할, 범위, 금지사항 문제는 `AGENTS.md` 보강으로 연결했는지 확인합니다.
- 반복 절차 문제는 skill 후보로 연결했는지 확인합니다.
- 외부 시스템 접근 문제는 MCP 계약으로 연결했는지 확인합니다.
- 승인, 차단, 감사 문제는 hook으로 연결했는지 확인합니다.
- 다음 작업에 남길 상태 문제는 memory contract로 연결했는지 확인합니다.
- 독립 검토나 병렬 조사 문제는 subagent 범위와 결과 검증으로 연결했는지 확인합니다.

### 5. HTML 분석 view

- `assets/harness-analysis-view.html`을 복사해 사용자 시나리오 데이터로 JSON 블록을 교체했는지 확인합니다.
- 사용자의 목적을 어떻게 이해했는지와 왜 에이전트가 목적대로 작동하지 않을 수 있는지를 줄글로 충분히 설명했는지 확인합니다.
- 상태전이 다이어그램을 보기 전에 상자, 화살표, 화살표 라벨, 먼저 볼 전이를 설명했는지 확인합니다.
- Mermaid가 단순 workflow chart가 아니라 상태전이 다이어그램인지 확인합니다.
- 상태전이 다이어그램의 화살표에 전이 조건과 작동 부품이 표시되는지 확인합니다.
- 부족 정보 때문에 사용자의 예상과 다르게 흘러가는 전이가 눈에 보이는지 확인합니다.
- 시나리오 흐름, 부족한 부분, 에이전트가 빠질 수 있는 상황, 사용자를 위한 질문이 모두 표시되는지 확인합니다.
- 실제 고객 자료, 인증정보, 개인정보가 HTML에 들어가지 않았는지 확인합니다.
- HTML view가 수강생이 바로 이해할 수 있는 쉬운 표현을 쓰는지 확인합니다.

## `AGENTS.md` 생성 체크리스트

### 1. Mission

- 에이전트가 맡는 일을 한 문장으로 썼는지 확인합니다.
- 맡지 않는 일과 위험한 자동화를 명시했는지 확인합니다.
- 최종 산출물이 아니라 초기 검증 산출물의 형식을 분리했는지 확인합니다.

### 2. Status

- `Stage`가 `idea`, `prototype`, `operational`, `core` 중 하나인지 확인합니다.
- `Output mode`가 실제 사용 방식과 맞는지 확인합니다.
- `Human review`가 필요한 위치를 숨기지 않았는지 확인합니다.

### 3. Inputs

- `Required`, `Optional`, `Sensitive`를 분리했는지 확인합니다.
- 필수 입력이 없을 때 질문으로 멈추는 규칙이 있는지 확인합니다.
- 민감 입력은 public repo에 저장하지 않는다고 적었는지 확인합니다.

### 4. Outputs

- 사용자가 바로 검토할 수 있는 산출물 이름으로 적었는지 확인합니다.
- 누락 정보, 검토표, 승인 대기 항목 같은 안전 산출물을 포함했는지 확인합니다.
- 외부 발송/게시 결과를 산출물로 오해하게 만들지 않았는지 확인합니다.

### 5. Operating Rules

- 숫자, 금액, 일정, 법적 판단, 공식 제출 내용은 근거 없는 추정을 금지했는지 확인합니다.
- 외부 API, 브라우저, 메일, 게시 도구 사용 전 승인 지점을 적었는지 확인합니다.
- 샘플/더미 데이터와 실제 운영 데이터를 분리했는지 확인합니다.
- 반드시 적용되어야 하는 규칙과 memory 후보를 분리했는지 확인합니다.

### 6. Suggested Workflow

- 첫 단계가 입력 확인 또는 범위 확인인지 확인합니다.
- workflow 상태 검증과 부족 정보 보고가 초안 생성보다 먼저 오는지 확인합니다.
- 초안 생성 전에 구조화/대응표/검증표를 만드는지 확인합니다.
- 마지막 단계가 사람 검토 또는 승인 대기인지 확인합니다.

### 7. Next Build Tasks

- 다음 작업이 2-4개 정도의 작은 단위인지 확인합니다.
- 첫 작업은 public-safe 샘플이나 템플릿으로 검증 가능한지 확인합니다.
- 외부 연동 작업은 정책 검토와 dry-run 뒤에 배치했는지 확인합니다.

### 8. Memory Contract

- 공식 Codex memory가 repo 안 `MEMORY.md`가 아니라 `~/.codex/memories/` 아래 생성 상태임을 전제했는지 확인합니다.
- 항상 적용되어야 하는 규칙을 `AGENTS.md`나 checked-in docs에 두었는지 확인합니다.
- memory 후보가 stable preference, recurring workflow, convention, known pitfall, 진행 상태 중 무엇인지 분류했는지 확인합니다.
- 저장 금지 데이터에 secrets, credentials, private originals, personal data가 포함되어 있는지 확인합니다.
- 저장 시점이 즉시가 아니라 eligible thread가 idle 상태가 된 뒤 background로 생성될 수 있음을 적었는지 확인합니다.
- 사용자가 `/memories` 또는 Codex 앱 설정으로 thread-level control을 할 수 있음을 적었는지 확인합니다.
- 공유 전 `~/.codex/memories/` 검토와 부적절한 기억 삭제/수정 절차를 적었는지 확인합니다.

## `.codex.toml` 생성 체크리스트

### 1. `[agent]`

- `id`가 폴더명과 일치하는지 확인합니다.
- `status`가 `AGENTS.md`의 Stage와 일치하는지 확인합니다.
- `visibility = "public-skeleton-only"`를 유지하는지 확인합니다.

### 2. `[scope]`

- `goal`이 실행 가능한 한 문장인지 확인합니다.
- `target_users`가 실제 검토자나 사용자와 맞는지 확인합니다.
- `non_goals`에 위험한 자동화와 과장된 범위를 넣었는지 확인합니다.

### 3. `[inputs]`

- `required`만으로 최소 동작이 가능한지 확인합니다.
- `optional`이 있으면 없을 때도 동작 방식이 있는지 확인합니다.
- `sensitive`에 인증, 개인정보, 내부 문서, 고객 정보가 빠지지 않았는지 확인합니다.

### 4. `[outputs]`

- `primary`가 초안, 검토표, 체크리스트 중심인지 확인합니다.
- `formats`가 현재 구현 단계와 맞는지 확인합니다.
- 미래 형식은 `hwp-future`, `pdf-future`처럼 구분했는지 확인합니다.

### 5. `[tools]`

- `required`는 로컬에서 검증 가능한 능력 위주인지 확인합니다.
- 외부 서비스는 `external_services`에 명시했는지 확인합니다.
- 아직 구현하지 않은 API는 `optional` 또는 `future` 성격으로 두었는지 확인합니다.

### 6. `[safety]`

- `human_review_required = true`인지 확인합니다.
- `external_actions_default = "dry_run"`인지 확인합니다.
- `approval_required_for`가 실제 위험 행동을 포함하는지 확인합니다.
- 결과물 전달 전 자동 체크포인트가 정의되어 있는지 확인합니다. 예: 근거 누락, 형식 오류, 확정 표현, 민감정보 포함 여부.
- `data_policy`가 커밋 금지 대상을 직접 말하는지 확인합니다.

### 7. `[workflow]`

- 단계가 입력 확인에서 검토 산출물까지 이어지는지 확인합니다.
- `validation`에 테스트 가능한 규칙이 들어 있는지 확인합니다.
- 불변식 위반 시 멈추거나 질문하는 경로가 있는지 확인합니다.
- 초안 작성 뒤 결과물 검토 체크포인트가 있고, 실패 시 수정 요청 또는 사람 검토로 돌아가는지 확인합니다.
- workflow 상태 모델링, 전이 검증, 부족 정보 영향 보고, 보강 계획 작성 단계가 있는지 확인합니다.

### 8. `[memory_contract]`

- `official_storage`가 `~/.codex/memories/`를 가리키는지 확인합니다.
- `repo_policy`가 실제 generated memories를 repo에 저장하지 않는다고 말하는지 확인합니다.
- `source_of_truth`가 `AGENTS.md`/docs와 Codex memories의 경계를 분리하는지 확인합니다.
- `must_define`에 기억 목적, 저장 금지, 저장 위치, 저장/사용 시점, thread control, 검토/삭제가 포함되는지 확인합니다.

## 완료 판정

새 에이전트 폴더는 아래 조건을 모두 만족해야 초안 완료로 봅니다.

- `AGENTS.md`와 `.codex.toml`이 서로 같은 mission, status, safety 정책을 말합니다.
- memory contract가 공식 Codex memory 동작과 source-of-truth 경계를 반영합니다.
- 하네스 부품 명세에서 시스템 프롬프트, 입력 프롬프트, 메모리, skill, MCP, subagent, hook의 입력/출력/실패 모드/불변식이 모두 채워져 있습니다.
- hook이 위험 행동 차단뿐 아니라 결과물 검토 체크포인트도 포함합니다.
- 결과물 검토 hook에는 trigger, 검토 기준, 통과/수정 필요/사람 검토 필요/차단 결과가 있습니다.
- workflow 상태 모델에서 각 핵심 상태의 입력, 출력, 다음 상태, 전이 조건이 채워져 있습니다.
- 상태전이 다이어그램이 있고 각 전이에 작동 부품과 판정이 표시되어 있습니다.
- 사용자의 예상과 다르게 흘러가는 전이가 다이어그램과 전이 해석에 표시되어 있습니다.
- 부족한 요청 상태가 시간 제한이나 단계 제한으로 바뀌지 않고 그대로 분석되었습니다.
- 부족 정보마다 약해지는 상태나 전이, 사용자에게 생길 문제, 보강할 하네스 부품이 연결되어 있습니다.
- 부족 정보마다 “에이전트가 빠질 수 있는 상황”이 쉬운 말로 설명되어 있습니다.
- 부족 정보가 hook, skill, `AGENTS.md` 규칙, 추가 정보 요청 중 무엇으로 이어지는지 “그래서 무엇을 보강해야 하는지” 문장으로 제안되어 있습니다.
