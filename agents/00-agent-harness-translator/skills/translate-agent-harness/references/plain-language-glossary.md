# 하네스 번역용 쉬운 용어 사전

이 문서는 0번 에이전트 하네스 번역 skill 안에서만 사용하는 용어 기준입니다. 0번 에이전트는 상위 폴더의 문서를 읽지 않고, 이 skill 폴더 안의 reference만 보고 수강생에게 설명합니다.

## 설명 원칙

- 비개발자 사회인이나 대학생도 이해할 수 있는 말로 설명합니다.
- 기술 용어를 먼저 말했으면 바로 쉬운 비유를 붙입니다.
- “무엇이 부족한가”보다 “그래서 업무 흐름에서 어떤 문제가 생기는가”를 먼저 이해시키는 방식으로 씁니다.
- 0번 에이전트는 파인만 학습식 강사처럼 행동합니다. 먼저 사용자의 업무 예시로 설명하고, 그다음 약속된 기술 용어를 붙입니다.
- 부족한 정보는 사용자의 실수가 아니라 에이전트가 길을 잃을 수 있는 빈칸으로 설명합니다.

## 핵심 용어

| 용어 | 쉬운 설명 |
|------|-----------|
| 에이전트 | 특정 반복 업무를 맡기기 위해 만든 작은 업무 담당자입니다. |
| 하네스 | 에이전트가 안전하게 일하도록 잡아주는 작업대입니다. 무엇을 읽고, 무엇을 만들고, 어디서 멈출지 정합니다. |
| `AGENTS.md` | 에이전트가 항상 지켜야 하는 업무 규칙표입니다. |
| `.codex.toml` | 에이전트의 설정표입니다. 이름, 목적, 입력, 출력, 도구, 승인 규칙을 정리합니다. |
| skill | 반복되는 작업 순서를 적어 둔 조리법입니다. |
| MCP | 외부 서비스나 파일을 안전하게 다루는 연결 장치입니다. |
| hook | 위험한 행동을 하기 전이나 결과물을 보내기 전에 잠깐 멈춰 확인하는 자동 검문소입니다. |
| subagent | 일을 나눠 맡기는 보조 담당자입니다. |
| memory | 다음에도 참고하면 좋은 안전한 기억입니다. 민감정보는 넣지 않습니다. |
| workflow | 일이 흘러가는 순서입니다. 접수, 확인, 초안, 검토, 전달 같은 단계입니다. |
| 전이 | 한 단계에서 다음 단계로 넘어가는 기준입니다. |
| gap | 사용자는 충분히 설명했다고 느꼈지만, 에이전트가 일하려면 더 필요한 빈칸입니다. |
| dry-run | 실제 발송, 업로드, 제출 없이 리허설만 해보는 방식입니다. |
| mock data | 실제 자료 대신 연습용으로 만든 가짜 데이터입니다. |

## 기술 용어를 붙이는 방식

기술 용어가 필요할 때는 아래 순서를 따릅니다.

1. 먼저 업무 예시로 말합니다.
2. 그다음 쉬운 설명을 붙입니다.
3. 마지막에 약속된 용어를 괄호로 표시합니다.

예:

```text
외부 제출 버튼을 누르기 전에는 사람이 한 번 멈춰서 확인해야 합니다.
또는 검토표를 보내기 전에 근거와 표현을 자동으로 점검해야 합니다.
이 멈춤 지점을 검문소처럼 두는 장치를 hook이라고 부릅니다.
```

## gap 설명 형식

0번 에이전트는 부족한 정보를 아래 형식으로 설명합니다.

```text
이 부분이 불분명합니다.
그래서 에이전트가 이런 상황에 빠질 수 있습니다.
이 부품을 보강하면 됩니다.
```

예:

```text
최종 제출 여부가 불분명합니다.
그래서 에이전트가 검토표만 만들어야 하는지, 실제 제출까지 해야 하는지 헷갈릴 수 있습니다.
이 부분은 AGENTS.md에 맡지 않는 일을 적고, hook에 제출 전 사람 승인 규칙과 결과물 전달 전 자동 점검 규칙을 두면 보강됩니다.
```
